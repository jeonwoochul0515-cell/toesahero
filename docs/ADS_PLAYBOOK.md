## 등록 중 겪은 함정 (반복 주의)

**`POST /ncc/keywords`는 실패를 HTTP 200 안에 숨긴다.** 배열로 838건을 보내 "성공 838"로 집계했는데 실제로는 727건만 등록됐다. 각 항목의 `resultStatus.code`를 검사해야 하며 `len(response)`를 성공 수로 세면 안 된다. 누락 111건의 원인은 `links` 형식 오류(평평한 문자열 → `{"pc":{"final":url}}` 중첩이어야 함). 재등록 시 107건 성공, 3건은 네이버 금지 키워드(`3909`)로 거부.

소재 생성은 헤드라인의 **가운뎃점(`·`)** 때문에 `1030` 오류로 실패 → 공백으로 대체해 통과. 상세는 전역 `~/.claude/CLAUDE.md` 검색광고 절 참조.

## 집행 시작 시각 (성과 계산의 기준선)

- 키워드 검수 승인 시각: **2026-07-20 17:56 KST** (`editTm` 2026-07-20T08:56:01Z, 내가 수정하지 않은 키워드 기준이라 검수 스탬프로 확정)

## 측정 — 2주 뒤 확인할 것

성과 조회는 API로 가능(`GET /stats?ids=<광고그룹ID>&fields=["impCnt","clkCnt","salesAmt","ctr","cpc"]&timeRange={"since":"...","until":"..."}`). 집계는 매시 정각 기준이라 당일 데이터는 늦게 반영된다.

판단 기준.
1. **그룹별 소진액** — 도구그룹이 2,000원을 매일 다 쓰는지, 상담그룹이 3,000원을 쓰는지. 상담그룹이 예산을 남기면 의뢰 의도 키워드의 검색량 자체가 부족하다는 뜻(월 3.3만회뿐)이라, 남는 예산을 도구그룹으로 옮기는 게 낫다.
2. **어떤 키워드가 클릭되는가** — 1,108개 중 실제로 클릭이 나는 건 소수일 것. 클릭 상위 키워드만 남기고 0클릭 키워드는 정리해도 되지만, **노출 안 되는 키워드는 비용이 0이라 굳이 삭제할 필요는 없다**(관리 편의 문제일 뿐).
3. **상담 신청이 늘었는가** — Firestore `consultations` 문서 수를 광고 전후로 비교. 전환 추적이 꺼져 있어 키워드 단위 연결은 아직 불가.
4. **도구그룹이 상담을 데려오는가** — 이번 구조의 핵심 가설은 "계산기를 쓰러 온 사람이 미지급 항목을 발견하면 상담으로 이어진다"이다. 2주 뒤 도구그룹 클릭 수 대비 상담 신청 증가분을 보고, 전혀 이어지지 않으면 도구그룹 예산을 줄여 상담그룹으로 옮길 것.
